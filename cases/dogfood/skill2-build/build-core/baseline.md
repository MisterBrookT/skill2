已按 `skill-creator` 规范判断：这个 skill 的 scope 应该是“管理 API 文档的工作流 agent skill”，不是“生成某一种 API 文档格式的模板库”。

**Scope 判断**

建议 skill 名称：`manage-api-docs`

适合包含：

- 盘点、创建、更新、重构 API 文档
- 对齐实现代码、OpenAPI/Swagger、SDK、README、示例请求
- 发现文档缺口、过期内容、破坏性变更
- 维护端点目录、认证说明、错误码、版本变更、示例
- 输出文档改动计划、补丁、审查报告或测试场景

暂不纳入核心 scope：

- 自动发布文档站点
- 从零设计完整 API 网关
- 生成 SDK
- 执行线上 API 请求做集成测试，除非用户明确允许
- 规定唯一文档格式，比如只支持 OpenAPI 或只支持 Markdown

# Manage API Docs

Use this skill to manage API documentation with implementation-aware checks and clear change control.

## Core Workflow

1. Determine the documentation source of truth.
   - Look for `openapi.yaml`, `openapi.json`, `swagger.*`, `docs/`, `README*`, route files, controller files, SDK clients, generated docs, or API tests.
   - If multiple sources exist, identify which one appears authoritative and which ones are derived or stale.
   - If unclear, report the ambiguity before making broad edits.

2. Identify the API surface.
   - For REST: methods, paths, path params, query params, headers, request bodies, response bodies, status codes, auth, pagination, rate limits, errors.
   - For GraphQL: schema, operations, input types, output types, auth, errors, deprecations.
   - For webhooks: event names, payloads, signing, retries, delivery guarantees, idempotency.
   - For SDK docs: public methods, parameters, return values, examples, exceptions, version compatibility.

3. Compare docs against implementation.
   - Prefer structured sources when available: OpenAPI specs, route manifests, schemas, validators, tests, type definitions.
   - Use code search before assuming behavior.
   - Treat examples as testable claims: URLs, fields, enum values, status codes, and error shapes should match actual behavior.
   - Flag implementation/documentation conflicts instead of silently inventing behavior.

4. Apply the smallest useful documentation change.
   - Preserve existing style, headings, terminology, and generated-file boundaries.
   - Do not edit generated docs unless the repo clearly expects generated output to be checked in.
   - If docs are generated from specs, update the source spec first.
   - Keep unrelated formatting churn out of the change.

5. Validate.
   - Run available doc/spec validation commands when present.
   - For OpenAPI, use the repo’s validator or existing scripts before introducing new tooling.
   - For Markdown-only docs, check links, examples, tables, and consistency manually if no validator exists.
   - If validation cannot be run, state why and what remains unverified.

## Before Editing

Check for these files and signals:

- `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `Makefile`, `.github/workflows/`
- `docs/`, `api/`, `openapi/`, `swagger/`, `spec/`
- `routes`, `controllers`, `handlers`, `schemas`, `validators`, `serializers`
- generated markers such as `DO NOT EDIT`, `generated`, `autogenerated`
- docs build tools such as Redocly, Swagger UI, Docusaurus, MkDocs, Mintlify, Stoplight, Speakeasy, Fern, Scalar

## Documentation Quality Checklist

A good API doc change should answer the relevant parts of this checklist:

- What endpoint, operation, event, or method is being documented?
- Who can call it and how authentication works.
- Required and optional inputs.
- Validation rules and default values.
- Success response shape and status code.
- Error response shape and common failure cases.
- Pagination, filtering, sorting, idempotency, or rate-limit behavior where applicable.
- Versioning, deprecation, or migration notes if behavior changed.
- Copy-pasteable examples that match the documented schema.
- Any security-sensitive details are described without leaking secrets.

## Conflict Handling

When sources disagree:

1. Prefer tests and implementation over prose docs.
2. Prefer schema validators and type definitions over examples.
3. Prefer OpenAPI/source specs over generated HTML or generated Markdown.
4. Prefer recent code and changelog entries over old docs.
5. If the correct behavior is still uncertain, surface the conflict clearly and ask before making behavior-specific claims.

## Output Modes

Choose the output format based on the user request:

- **Audit**: list findings ordered by severity with file references and suggested fixes.
- **Patch**: make focused documentation/spec edits and report validation results.
- **Plan**: propose doc structure, ownership, migration steps, and validation strategy.
- **Review**: prioritize incorrect, stale, ambiguous, missing, or risky documentation.
- **Scaffold**: create a minimal API documentation structure only when the user asks for new docs.

## Style Rules

- Match the repo’s existing documentation voice.
- Prefer concrete examples over abstract descriptions.
- Avoid claiming runtime behavior without evidence.
- Keep endpoint docs scannable: method/path, purpose, auth, params, request, response, errors, examples.
- Use consistent terminology for users, accounts, organizations, projects, tokens, keys, and resources.
- Keep public API docs user-facing; keep implementation details out unless they affect API behavior.

## Safety Rules

- Do not call live production APIs unless the user explicitly asks and provides permission.
- Do not expose real secrets, tokens, credentials, customer data, or internal URLs in examples.
- Replace sensitive values with realistic placeholders.
- Do not rewrite large documentation sets without first identifying generation rules and source of truth.

**测试场景**

1. **OpenAPI 与代码不一致**
   - 用户请求：`检查 API 文档是否和实现一致，不要改代码。`
   - 仓库情况：`openapi.yaml` 中 `GET /users/{id}` 返回 `email`，实现实际还返回 `status`。
   - 期望行为：agent 找到 spec、路由和响应 schema，指出不一致，建议更新 OpenAPI 或确认字段是否公开，不直接猜测。

2. **更新新增 endpoint 文档**
   - 用户请求：`帮我给新增的 POST /invoices 写文档。`
   - 仓库情况：已有 Markdown endpoint 文档和 controller/schema。
   - 期望行为：agent 读取现有文档风格，补齐认证、请求体、响应、错误码和示例，保持格式一致。

3. **识别生成文件**
   - 用户请求：`修一下 docs/api.md 里的参数说明。`
   - 仓库情况：`docs/api.md` 顶部有 `DO NOT EDIT`，源文件是 `openapi.yaml`。
   - 期望行为：agent 不直接改生成文件，转而建议或修改源 spec。

4. **API 文档审查**
   - 用户请求：`review 这次 API docs 改动。`
   - 仓库情况：PR 修改了错误码表和几个 endpoint 示例。
   - 期望行为：按严重程度列出问题，例如示例字段不存在、状态码和实现冲突、认证说明遗漏，附文件行号。

5. **Webhook 文档补全**
   - 用户请求：`补全 webhook 文档，尤其是签名和重试。`
   - 仓库情况：代码里有 webhook sender、签名 helper、retry policy。
   - 期望行为：agent 提取事件名、payload、签名头、重试策略、幂等建议，避免泄露真实密钥。

6. **多来源冲突**
   - 用户请求：`把 API README 更新成最新。`
   - 仓库情况：README、OpenAPI、SDK docs 三处描述不同。
   - 期望行为：agent 先判断 source of truth，明确冲突来源；如果无法确认，先输出冲突清单和建议决策点。
